# DevWiki MVP Goal

## 목적

DevWiki의 첫 번째 실사용 목표는 개발자들이 백엔드 기술 면접 지식을 함께 작성, 수정, 탐색할 수 있는 협업형 위키를 완성하는 것이다.

이 Goal은 단순히 화면을 만드는 것이 아니라, Supabase Auth/Postgres/Storage와 Next.js 앱이 연결된 상태에서 아래 기능들이 실제 사용자 흐름으로 동작하는 것을 완료 기준으로 삼는다.

## 전제 조건

- `.env.local`에 Supabase URL과 publishable key가 설정되어 있다.
- `.env.local`에 서버 전용 `SUPABASE_SERVICE_ROLE_KEY`가 설정되어 있다.
- `.env.local`에 활성 멤버 이메일인 `DEVWIKI_E2E_EMAIL`이 설정되어 있다.
- `.env.local`에 해당 멤버 Auth user의 `DEVWIKI_E2E_PASSWORD`가 설정되어 있다.
- Supabase SQL Editor에서 초기 마이그레이션이 적용되어 있다.
- Supabase SQL Editor에서 추가 마이그레이션이 파일명 순서대로 모두 적용되어 있다.
- `public.members`에 최소 1명 이상의 활성 멤버 이메일이 등록되어 있다.
- Supabase Auth에 멤버 이메일과 비밀번호가 등록되어 있다.

## 범위

1. 로그인
2. 문서 목록
3. 문서 작성/수정
4. Markdown 미리보기
5. Mermaid 다이어그램 렌더링
6. 문서 변경 이력
7. 태그/검색
8. 이미지 업로드

## 완료 기준

### 1. 로그인

목적: 등록된 멤버만 DevWiki에 접근하고 문서를 작성/수정할 수 있게 한다.

검증:
- 등록된 이메일과 비밀번호로 로그인할 수 있다.
- 로그인 성공 시 DevWiki 세션이 생성되고 문서 목록으로 이동한다.
- 등록되지 않은 이메일은 로그인 후에도 문서 접근 또는 변경이 차단된다.
- 로그아웃 후 작성/수정 페이지 접근 시 로그인 페이지로 이동한다.

### 2. 문서 목록

목적: 멤버가 현재 정리된 기술 개념을 빠르게 훑고 들어갈 수 있게 한다.

검증:
- 로그인한 사용자는 Supabase에 저장된 문서 목록을 볼 수 있다.
- 각 문서는 제목, 요약, 상태, 태그, 마지막 수정일을 보여준다.
- 문서가 없을 때 빈 상태 UI가 표시된다.
- Supabase 연결 전 데모 모드는 실제 연결 후 사라진다.

### 3. 문서 작성/수정

목적: 멤버가 기술 면접 개념 문서를 자유롭게 생성하고 계속 보강할 수 있게 한다.

검증:
- 새 문서를 제목, slug, 요약, 태그, 본문, 상태와 함께 저장할 수 있다.
- 기존 문서를 수정하면 변경된 제목, 요약, 본문, 태그, 상태가 반영된다.
- slug가 비어 있으면 제목 기반으로 자동 생성된다.
- 중복 slug는 자동으로 충돌을 피한다.
- 권한 없는 사용자는 생성/수정 Server Action을 직접 호출해도 실패한다.

### 4. Markdown 미리보기

목적: 작성자가 저장 전 렌더링 결과를 바로 확인할 수 있게 한다.

검증:
- 편집, 분할, 미리보기 모드 전환이 동작한다.
- 제목, 목록, 코드블록, 표, 링크가 Markdown 문법대로 렌더링된다.
- 긴 문서에서도 에디터와 미리보기 영역이 깨지지 않는다.
- 저장되는 원문은 Markdown 형식으로 유지된다.

### 5. Mermaid 다이어그램 렌더링

목적: 멱등성, 트랜잭션, 메시지 큐 같은 개념을 시각적으로 설명할 수 있게 한다.

검증:
- ` ```mermaid ` 코드블록이 SVG 다이어그램으로 렌더링된다.
- sequenceDiagram, flowchart 같은 기본 Mermaid 문법이 동작한다.
- Mermaid 문법 오류가 있을 때 페이지 전체가 깨지지 않고 오류 메시지를 보여준다.
- 상세 페이지와 미리보기 양쪽에서 동일하게 렌더링된다.

### 6. 문서 변경 이력

목적: 누가 어떤 의도로 문서를 바꿨는지 추적할 수 있게 한다.

검증:
- 문서 생성 시 첫 revision이 생성된다.
- 문서 수정 시 새 revision이 생성된다.
- 상세 페이지에서 최근 변경 이력을 볼 수 있다.
- revision에는 제목, 본문, 요약, 수정 요약, 수정자, 수정 시간이 남는다.

### 7. 태그/검색

목적: 개념들이 쌓였을 때 관련 문서를 빠르게 찾고 묶을 수 있게 한다.

검증:
- 문서 저장 시 쉼표로 입력한 태그가 생성/연결된다.
- 문서 수정 시 태그 연결이 최신 입력값으로 갱신된다.
- 목록에서 태그가 표시된다.
- 검색어로 제목, 요약, 태그 기반 필터링이 가능하다.
- 검색 결과가 없을 때 빈 상태 UI가 표시된다.

### 8. 이미지 업로드

목적: 면접 개념 설명에 필요한 구조도, 캡처, 손그림 같은 시각 자료를 문서에 첨부할 수 있게 한다.

검증:
- 로그인한 멤버는 이미지 파일을 업로드할 수 있다.
- 업로드 대상은 `devwiki-assets` Supabase Storage bucket이다.
- 허용 MIME 타입은 png, jpeg, webp, gif로 제한된다.
- 업로드 후 Markdown 이미지 문법으로 문서에 삽입할 수 있다.
- 비로그인 사용자와 비멤버는 업로드할 수 없다.
- 업로드된 이미지는 문서 상세 페이지에서 정상 렌더링된다.

## 최종 검증 시나리오

1. Supabase에 테스트 멤버 이메일을 등록한다.
2. 해당 이메일과 비밀번호로 로그인한다.
3. `멱등성 테스트` 문서를 작성한다.
4. 본문에 Markdown 표, 코드블록, Mermaid 다이어그램을 넣는다.
5. 이미지 하나를 업로드하고 본문에 삽입한다.
6. 문서를 저장하고 상세 페이지에서 모든 콘텐츠가 렌더링되는지 확인한다.
7. 문서를 수정하고 수정 요약을 입력한다.
8. 변경 이력에 생성/수정 revision이 남는지 확인한다.
9. 태그와 검색어로 문서를 다시 찾는다.
10. 로그아웃 후 문서 작성/수정/업로드가 차단되는지 확인한다.

## 완료 판정

아래 명령과 최종 E2E gate가 모두 통과하면 Goal을 완료로 본다.

```bash
npm run lint
npm run build
npm run verify:supabase
npm run verify:mvp
```

`npm run verify:mvp`는 다음 검증을 포함한다.

- Supabase readiness 검증
- 인증된 멤버 데이터 흐름 검증
- 비멤버/비로그인 차단 검증
- 브라우저 기반 password login, `/`, `/documents/new`, `/documents/[slug]`, `/documents/[slug]/edit` 검증
- Markdown, Mermaid, 이미지 업로드/렌더링 검증
- revision, 태그, 검색 검증

잔여 리스크:
- Supabase Auth redirect URL 설정이 배포 환경과 일치하지 않으면 로그인 콜백이 실패할 수 있다.
- CI/CD 배포 환경변수는 로컬 `.env.local`과 별도로 설정해야 한다.
- 동시 편집 충돌 처리는 MVP 범위에 포함하지 않는다.
